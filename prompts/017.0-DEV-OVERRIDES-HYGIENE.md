# 017.0-DEV-OVERRIDES-HYGIENE: Overrides Hygiene — Surface Stale and Vulnerable Override Pins

## Release Goal

**Release 0.11: Overrides Hygiene**

Surface entries in the `package.json` `overrides` block that are stale (behind the latest published version) and/or vulnerable (within a current `npm audit` advisory range), so that override pins added as historical security measures do not silently rot into the very risk they were meant to mitigate. Default on across all output formats so override-hygiene findings reach the maintainer's eye without flag-juggling, mirroring the ADR-0018 unfixable-surface default-on precedent.

## How This Story Contributes

`package.json` `overrides` are a common security-hardening lever — a project pins a (typically transitive) package to a specific safe version to dodge a known advisory or force a parent-version bump. The pin is a snapshot of "safe today", not "safe forever". As new advisories are disclosed and as the broader ecosystem moves on, the override target itself drifts: it falls behind latest, and frequently it later falls inside a new advisory range. The override that once meant "secure" then means "vulnerable", and as a hard pin it actively blocks the natural upgrade path that would otherwise fix the issue.

Today `dry-aged-deps` is blind to this whole class of self-inflicted, project-controlled risk. Its analysis pipeline reads `npm outdated --json`, `dependencies` / `devDependencies` for prod/dev classification, and `npm audit --json` for vulnerabilities — but nothing parses `overrides`. The brace-expansion incident observed 2026-05-25 on this repo is the originating example: a `"brace-expansion": "^4.0.1"` override added 2025-11-18 as a security pin (commit `4664227`) silently rotted into the GHSA-f886-m6hf-6m8v range without surfacing anywhere in the tool's output.

This story adds a new `overrides-hygiene` module that parses the `overrides` block, ages each pin via the existing `fetchVersionTimes()` / `calculateAgeInDays()` infrastructure, cross-references each pin against `npm audit` advisory ranges and `npm outdated` data, and emits a structured findings section across table / JSON / XML outputs. The new section reaches every output surface so retroactive-vet pipelines and audit consumers pick it up without bespoke integration.

This is **gap #1** of two surfaced by P013. The companion **gap #2** (sharpening the unfixable-surface reason logic to be `fixAvailable`-aware so override-fixable transitive vulns aren't mislabelled "vulnerable transitive dependency") amends ADR-0018 separately and is **out of scope here** per RFC-001's explicit Scope/Out-of-Scope split.

## User Story

**Format**: So that I can catch override pins that have rotted into the very risk they were meant to mitigate, as a project maintainer, I want `dry-aged-deps --check` to list `package.json` `overrides` entries that are stale and/or fall inside a current advisory range, by default, in every output format.

**INVEST Criteria Compliance**:

- **Independent**: Builds on existing version-time fetching (`src/build-rows.js`) and vulnerability checking (`src/check-vulnerabilities.js`); does not require new external tooling.
- **Negotiable**: Per-call opt-out via `--no-overrides-hygiene`; default-on policy is reviewable in config.
- **Valuable**: Closes a documented self-inflicted-risk gap (P013) with a live observed instance (brace-expansion override on this repo). Extends the tool's core "see safe updates" surface to a class it was previously blind to.
- **Estimable**: Reuses existing version-time and audit-cross-reference machinery; new code is the parser, the cross-reference, and per-format render surfaces. Single iteration.
- **Small**: One new module, three formatter extensions, one CLI flag, one config key. No new external dependency, no new vendor relationship.
- **Testable**: Detection rule is mechanical (parse override → age via registry → cross-reference advisory range + npm outdated). Each output format gets its own snapshot test; live-case regression test pins the original brace-expansion scenario.

## Acceptance Criteria

- [ ] **REQ-OVERRIDES-PARSE**: Parse `package.json` `overrides` in both shapes per the npm overrides spec — simple (`"name": "version"`) and nested (`"name": { ".": "version", "sub-dep": "version" }`). Unknown / malformed entries surface as a warning row, not a hard error.
- [ ] **REQ-OVERRIDES-AGE**: For each override entry, fetch the registry publish time for the pinned version via existing `fetchVersionTimes()` infrastructure and compute days-since-publish via existing `calculateAgeInDays()`. Pins to a non-existent version surface as a warning row.
- [ ] **REQ-OVERRIDES-AUDIT-XREF**: For every override pin that falls within a current advisory range from `npm audit --json`, surface the advisory ID (e.g. `GHSA-xxxx`), severity band, and patched-version range. Reuses existing `checkVulnerabilities()` infrastructure.
- [ ] **REQ-OVERRIDES-OUTDATED-XREF**: For every override pin that is behind the latest published version, surface the latest version and the lag (days plus semver delta — major / minor / patch).
- [ ] **REQ-OVERRIDES-TABLE**: Table output appends an `Override hygiene` section after the existing outdated table (and after the unfixable section from ADR-0018 if present). Columns: `Override`, `Pinned`, `Latest`, `Age`, `Reason`, `Safe Upgrade`. Existing columns and sections are unchanged.
- [ ] **REQ-OVERRIDES-JSON**: JSON output gains a top-level `overridesHygiene` array of `{ name, pinned, latest, ageDays, reason, advisories: [{ id, severity, patchedRange }], safeUpgrade }` objects. Existing `rows` / `packages` / `unfixable` (from ADR-0018) fields are unchanged.
- [ ] **REQ-OVERRIDES-XML**: XML output gains a top-level `<overridesHygiene>` element with one `<override>` child per row, mirroring the JSON shape.
- [ ] **REQ-OVERRIDES-DEFAULT-ON**: The surface is on by default in all output formats. `--no-overrides-hygiene` (CLI) or `"overrides-hygiene": false` (config) suppresses it across all formats. Mirrors the ADR-0018 default-on precedent.
- [ ] **REQ-OVERRIDES-EXCEPTION-RESPECT**: Advisories in `audit-resolve.json` exclusions are filtered out of override-hygiene findings on the audit-cross-reference axis (i.e. an excepted advisory does not produce a vulnerable-override row). Stale-only findings (behind-latest, no advisory hit) are not gated by `audit-resolve.json`. Mirrors ADR-0008 + ADR-0018 exception-respect symmetry.
- [x] **REQ-OVERRIDES-EXIT-CODE**: Exit codes are unchanged from ADR-0003 / ADR-0004 except for one narrow extension: `--check` exits 1 when one or more override pins have a **safe upgrade target available** (i.e. a later version exists that satisfies the age threshold and is not within an advisory range). Override pins that are stale and/or vulnerable but have **no safe upgrade target** surface informationally and do NOT widen `--check`'s exit-1 trigger — matching the ADR-0018 "informational, no exit-code shift" pattern. (2026-06-02: RFC-001 T6 — bin/dry-aged-deps.js extended exit-1 trigger to OR `summary.overridesWithSafeUpgrade > 0` against `summary.safeUpdates > 0`; same logic applied to default-mode exit code per ADR-0003 § "default and --check exit semantics are equivalent" implementation parity. ADR-0004 Confirmation §3 references literal `summary.safeUpdates` — the RFC is the load-bearing amendment surface per ADR-0019.)
- [ ] **REQ-OVERRIDES-AUDIT-ARTEFACT**: Override-hygiene findings appear in JSON and XML outputs (not table-only) so retroactive-revet pipelines, audit consumers, and the auto-update workflow's PR-body composition surface (ADR-0017) pick the findings up without bespoke integration. The PR-body section is added in REQ-OVERRIDES-WORKFLOW below.
- [ ] **REQ-OVERRIDES-WORKFLOW**: `.github/workflows/auto-update.yml` reads the new `overridesHygiene` field from `updates.json` and adds a `## Override hygiene` section to the PR body on both green and budget-exhaust paths, mirroring the ADR-0017 + ADR-0018 workflow integration precedent.
- [ ] **REQ-OVERRIDES-SCHEMA-COMPAT**: The JSON and XML schemas remain backward-compatible — existing consumers that ignore unknown top-level fields continue to operate unchanged. The output section is additive; no existing field shape or value changes.
- [ ] **REQ-OVERRIDES-REASON-TAXONOMY**: Each finding's `reason` field uses a fixed short vocabulary so consumers can pattern-match: `stale: <N> days behind latest`, `vulnerable: <advisory-id>`, `stale-and-vulnerable: <N> days behind, <advisory-id>`, `pin-to-unknown-version`, `pin-to-malformed-entry`. Mirrors the ADR-0018 short-reason-vocabulary pattern.

## Requirements (Current Implementation or To Be Implemented)

- **REQ-OVERRIDES-PARSE**: New module `src/overrides-hygiene.js` exports a parser that accepts a loaded `package.json` object and returns a normalised array of override entries with `{ name, pinned, isSimpleShape }` plus a separate array of malformed entries.
- **REQ-OVERRIDES-AGE**: `src/overrides-hygiene.js` calls existing `fetchVersionTimes()` from `src/build-rows.js` for each override target version. Pins to non-existent versions surface as `pin-to-unknown-version` rows.
- **REQ-OVERRIDES-AUDIT-XREF**: `src/overrides-hygiene.js` consumes the already-fetched `npm audit --json` payload from `src/check-vulnerabilities.js` (passed in — does not re-invoke `npm audit`). Cross-references each pinned version against advisory `vulnerable_versions` ranges.
- **REQ-OVERRIDES-OUTDATED-XREF**: `src/overrides-hygiene.js` consumes the already-fetched `npm outdated --json` payload from the existing pipeline (passed in — does not re-invoke `npm outdated`). For overrides on packages not present in `npm outdated`, fetch latest via the registry helper used by `fetchVersionTimes()`.
- **REQ-OVERRIDES-PIPELINE-WIRE**: `src/print-outdated.js` invokes `runOverridesHygiene()` after `applyFilters` and before output formatter dispatch, mirroring the existing data-flow seam. The result rides alongside `rows` / `unfixable` into each formatter.
- **REQ-OVERRIDES-TABLE-RENDER**: Extend `src/output-utils.js` / `src/print-utils.js` to render the new `Override hygiene` section after the outdated table (and after the unfixable section if rendered).
- **REQ-OVERRIDES-JSON-RENDER**: Extend `src/json-formatter.js` to emit the `overridesHygiene` array.
- **REQ-OVERRIDES-XML-RENDER**: Extend `src/xml-formatter.js` / `xml-formatter-utils.js` to emit the `<overridesHygiene>` element.
- **REQ-OVERRIDES-CLI-FLAGS**: Add `--no-overrides-hygiene` parsing in `src/cli-options.js` / `src/cli-parser-utils.js`.
- **REQ-OVERRIDES-CONFIG-KEY**: Accept `"overrides-hygiene"` (boolean) key in `.dry-aged-deps.json` via `src/config-loader.js`.
- **REQ-OVERRIDES-EXCEPTION-LOOKUP**: Cross-reference advisory IDs against `audit-resolve.json` exclusions before adding a vulnerable-override row, reusing the helper introduced for ADR-0018.
- **REQ-OVERRIDES-EXIT-CODE-LOGIC**: Extend the exit-code computation in the appropriate orchestrator to fire exit-1 when one or more override-hygiene findings carry a non-null `safeUpgrade` field. Pre-existing exit-1 triggers (safe direct-dependency updates available) remain unchanged — the new trigger is additive within the existing "safe path forward exists" semantic. (2026-06-02: implemented at `src/print-outdated.js` via `countOverridesWithSafeUpgrade()` helper threaded into the returned `summary.overridesWithSafeUpgrade` field — populated in both `handleNoOutdated` and the main path. `bin/dry-aged-deps.js` ORs `summary.overridesWithSafeUpgrade > 0` into the existing `summary.safeUpdates > 0` exit-1 condition.)
- **REQ-OVERRIDES-WORKFLOW-INTEGRATION**: `.github/workflows/auto-update.yml` "Open pull request" step reads `overridesHygiene` from `updates.json` and composes the new section into the PR body.
- **REQ-OVERRIDES-REGRESSION-TEST**: Live-case regression test exercises the original brace-expansion scenario from P013. Fixture: a `package.json` with a stale override pin (e.g. `"brace-expansion": "^4.0.1"`) plus a current advisory hitting that range. Expected output: a `vulnerable: GHSA-...` finding with a non-null `safeUpgrade` field; no row labelled "vulnerable transitive dependency" (gap #2's surface is owned by a separate ADR-0018 amendment).

## Story Notes

**Gall's Law Compliance**: Adds a new module + a new section in three output paths + one CLI flag + one config key. Reuses existing version-time, audit-cross-reference, exception-lookup, and output-formatter machinery. No new external dependency, no new vendor relationship, no new authentication surface.

**Non-breaking nature**: The new output section is **additive across every surface**. Table output gains a new appended section but does not change any existing column or row. JSON gains a new top-level `overridesHygiene` field — existing `rows` / `packages` / `unfixable` fields and their shapes are unchanged; consumers that ignore unknown top-level fields continue to operate unchanged. XML gains a new top-level element with the same backward-compatibility profile. Exit codes are unchanged for any existing scenario; the new exit-1 trigger fires only on a new class of finding (override pin with safe upgrade target) within the existing ADR-0003 "safe path forward exists" semantic. **This is not a breaking change**; no major-version bump is required. The changelog entry SHOULD call out the new default-on surface so consumers know to expect new output rows.

**Boundary with gap #2 (`fixAvailable`-aware unfixable reason)**: This spec adds a new module that surfaces overrides hygiene. It does NOT modify `src/find-unfixable-vulns.js` or the unfixable-surface reason logic. The mislabel symptom from P013 ("vulnerable transitive dependency" applied to override-fixable vulns) remains until the gap-#2 amendment to ADR-0018 ships separately. The two workstreams are independently shippable.

**Boundary with auto-update workflow**: The auto-update workflow PR-body extension is part of this spec (REQ-OVERRIDES-WORKFLOW) — without it, the override-hygiene surface only reaches a maintainer running `dry-aged-deps` locally. The PR body is the durable audit surface per the project's audit-trail discipline and must carry the same finding class.

**Exit-code rationale**: The narrow exit-1 contract (fire only when an override pin has a safe upgrade target available) aligns with ADR-0003's "safe updates available" semantic. A stale-but-no-safe-target override surfaces informationally exactly like an ADR-0018 unfixable row — the user sees it but `--check` does not block CI. This preserves the existing `npm run check:deps` gate behaviour for the common case and only extends the failure surface to actionable findings.

**Exception-respect rationale (REQ-OVERRIDES-EXCEPTION-RESPECT)**: An advisory excepted in `audit-resolve.json` is the maintainer's explicit "known and accepted" mark — surfacing it again as an override-hygiene vulnerable row would re-surface noise the maintainer has already silenced. The stale-only axis is not gated by `audit-resolve.json` because staleness has no per-advisory exception path; staleness is judged purely on the registry vs. the pinned version.

**Reason-taxonomy rationale (REQ-OVERRIDES-REASON-TAXONOMY)**: Mirrors the ADR-0018 short-reason-vocabulary pattern so downstream consumers (`updates.json` parsers, workflow PR-body composition, audit pipelines) can pattern-match on a small fixed set rather than parsing free text.

**Unratified-dependency posture (accepted risk)**: RFC-001 §Related explicitly records that ADR-0018 / ADR-0019 are currently `.proposed.md` (lack `human-oversight: confirmed` frontmatter) and that the RFC accepts this risk rather than blocking. The same posture extends to this spec — the cited ADRs and JTBDs are descriptive cites for boundary-setting, not load-bearing on the spec's correctness. If the cited governance is materially amended at canonical confirmation, the spec is reviewable for impact at that point. The wider JTBD ratification question (JTBD-001, JTBD-006, JTBD-204, `project-maintainer` persona, `tech-lead` persona) is queued for the next interactive session.
