# 016.0-DEV-SURFACE-UNFIXABLE-VULNERABILITIES: Surface Known-Vulnerable-But-Unfixable Packages

## Release Goal

**Release 0.10: Vulnerability Visibility**

Surface packages that carry a known vulnerability AND have no available version satisfying both the vuln fix and the maturity/security thresholds. Default on across all output formats so unfixable vulns reach the maintainer's eye without flag-juggling.

## How This Story Contributes

Today, vulnerabilities only surface in `dry-aged-deps` output when there IS a safe update available (the row shows the safe target along with severity). Vulnerable packages with no safe path forward are invisible — they don't appear in `--check`, never reach the auto-update workflow's PR body, and are only visible if the maintainer happens to scroll through the prepush audit log. The brace-expansion advisory observed 2026-05-18 (GHSA-f886-m6hf-6m8v, moderate, no safe-version path in the current transitive chain) is a concrete example of the missed-signal class.

This story adds a complementary surface so unfixable vulns enter the maintainer's normal review loop instead of being noise in a CI log.

## User Story

**Format**: So that I can plan mitigations for vulnerabilities the autonomous loop cannot fix, as a developer, I want `dry-aged-deps --check` to list known-vulnerable packages that have no safe-version path, by default, in every output format.

**INVEST Criteria Compliance**:

- **Independent**: Builds on existing vulnerability filtering (story 004) and smart-search filter pipeline (ADR-0014); does not require new external tooling.
- **Negotiable**: Severity floor default (all severities vs moderate-and-above) is configurable per-call and per-project.
- **Valuable**: Closes a visibility gap that directly motivates the autonomous-loop design — vulns that cannot be fixed autonomously must still reach the maintainer.
- **Estimable**: Reuses npm-audit-json parsing and the smart-search filter pipeline; new code is detection + new section in each formatter + workflow PR-body extension.
- **Small**: Single iteration. No new external dependency.
- **Testable**: Detection rule is mechanical (advisory match + smart-search absence + exception cross-check); each output format gets its own snapshot test.

## Acceptance Criteria

- [ ] **REQ-UNFIXABLE-DETECT**: Identify packages that (a) have a vuln per `npm audit --json`, (b) have no installable version satisfying the ADR-0014 smart-search filter pipeline, and (c) are not listed in `audit-resolve.json` exclusions.
- [ ] **REQ-UNFIXABLE-TABLE**: Table output appends a `Known vulnerabilities without safe fix` section after the existing outdated table. Columns: `Name`, `Severity`, `Advisory`, `Reason`. Existing outdated-table columns are unchanged.
- [ ] **REQ-UNFIXABLE-JSON**: JSON output gains a top-level `unfixable` array of `{ name, severity, advisory, reason, via }` objects. Existing `rows` / `packages` fields are unchanged.
- [ ] **REQ-UNFIXABLE-XML**: XML output gains a top-level `<unfixable>` element with one `<vulnerability>` child per row.
- [ ] **REQ-UNFIXABLE-DEFAULT-ON**: The surface is on by default in all output formats. `--no-unfixable` (CLI) or `"unfixable": false` (config) suppresses it across all formats.
- [ ] **REQ-UNFIXABLE-SEVERITY-FLOOR**: All severities (low / moderate / high / critical) are surfaced by default. `--unfixable-level=<low|moderate|high|critical>` or `"unfixable-level": "<level>"` raises the floor.
- [ ] **REQ-UNFIXABLE-FIXABLE-SOON-BOUNDARY**: A package with a patched version that is too young to meet `--min-age` is NOT classified as unfixable (it is fixable-soon and surfaces via existing logic).
- [ ] **REQ-UNFIXABLE-EXCEPTION-RESPECT**: Advisories in `audit-resolve.json` exclusions are filtered out of the unfixable surface regardless of severity.
- [ ] **REQ-UNFIXABLE-EXIT-CODE**: Exit codes are unchanged from ADR-0003 / ADR-0004. A `--check` run with no safe updates but with unfixable vulns still exits 0.
- [ ] **REQ-UNFIXABLE-WORKFLOW**: `.github/workflows/auto-update.yml` reads the new `unfixable` field from `updates.json` and adds a `## Known vulnerabilities without safe fix` section to the PR body on both green and budget-exhaust paths.
- [ ] **REQ-UNFIXABLE-SCHEMA-COMPAT**: The JSON and XML schemas remain backward-compatible — existing consumers that ignore unknown fields continue to operate unchanged.

## Requirements (Current Implementation or To Be Implemented)

- **REQ-UNFIXABLE-DETECT**: New module (or extension of `src/check-vulnerabilities.js` and `src/apply-filters.js`) that produces a typed `unfixable` array alongside the existing `rows` output.
- **REQ-UNFIXABLE-TABLE**: Extend `src/output-utils.js` / `src/print-utils.js` to render the new section after the outdated table.
- **REQ-UNFIXABLE-JSON**: Extend `src/json-formatter.js` to emit the new array.
- **REQ-UNFIXABLE-XML**: Extend `src/xml-formatter.js` (and `xml-formatter-utils.js`) to emit the new element.
- **REQ-UNFIXABLE-CLI-FLAGS**: Add `--no-unfixable` and `--unfixable-level=<level>` parsing in `src/cli-options.js` / `src/cli-parser-utils.js`.
- **REQ-UNFIXABLE-CONFIG-KEYS**: Accept `unfixable` (boolean) and `unfixable-level` (string) keys in `.dry-aged-deps.json` via `src/config-loader.js`.
- **REQ-UNFIXABLE-EXCEPTION-LOOKUP**: Cross-reference advisory ids against `audit-resolve.json` exclusions before adding a row to the unfixable set.
- **REQ-UNFIXABLE-WORKFLOW-INTEGRATION**: `.github/workflows/auto-update.yml` "Open pull request" step reads `unfixable` from `updates.json` and formats the new section into the PR body.

## Dependencies

- **004.0-DEV-FILTER-VULNERABLE-VERSIONS**: Vulnerability detection machinery (npm audit + version cross-reference).
- **ADR-0014**: Smart-search filter pipeline whose absence-of-output is the unfixable signal.
- **ADR-0008**: `audit-resolve.json` exception conventions.

- **ADR-0018**: Architectural decision documenting the design.

## Story Notes

**Gall's Law Compliance**: Adds a new section to existing output paths; reuses existing detection and filter machinery. No new external dependency, no new vendor relationship, no new authentication surface.

**Boundary with fixable-soon**: This story is strictly about packages where NO version satisfies the smart-search filter pipeline. If a patched version exists but is younger than `--min-age`, that package is fixable-soon — covered by existing logic, out of scope here. The two surfaces never overlap.

**Exit-code preservation**: The new surface is informational. `--check` exit-1 trigger is unchanged (safe updates available). A repo with zero safe updates and ten unfixable vulns still exits 0 from `--check`, so the existing `npm run check:deps` gate behaviour does not shift.

**Workflow integration boundary**: The workflow PR-body extension is part of this story — without it, the unfixable surface only reaches a maintainer running `dry-aged-deps` locally. The PR body is the durable audit surface per JTBD-008 and must carry the same information.

**Severity-floor default rationale**: The brace-expansion advisory that motivated this story is moderate. Defaulting to "all severities" honours the spirit of "the safe path is the default path" — anything quieter would have missed the originating example. JTBD-006's "low cognitive cost" tension is acknowledged in ADR-0018's Bad consequences with a reassessment trigger.
