# 018.0-DEV-EXPOSURE-AWARE-SOAK: Exposure-Aware Soak — Severity-Scaled Soak Modifier

## Release Goal

**Release 0.12: Exposure-Aware Soak**

Replace `dry-aged-deps`' unconditional age soak with an opt-in, exposure-aware, severity-scaled modifier: when the project is currently exposed (per `npm audit --json`) to a Critical or High vulnerability in a package whose fix lands in a fresher-than-soak release, the effective soak shortens proportional to that exposure severity. Critical exposure drops the soak to a 0-day floor (pull the fix immediately); High exposure halves the soak; Moderate / Low / None preserve the operator's `min-age` unchanged. Ships behind `--exposure-aware-soak` opt-in flag, **default-OFF for v1**, so existing non-opt-in users see exactly the pre-RFC-002 soak behaviour. Graduation to opt-out is a v2 decision requiring its own ADR and changelog entry.

## How This Story Contributes

`dry-aged-deps`' core value is a risk trade: it waits `min-age` days before recommending an update, so a freshly-published release has time for the ecosystem to surface supply-chain compromises or unvetted regressions. Time-on-registry is the cheap proxy for "this release has been vetted."

But that soak is currently **absolute**. It does not account for the other side of the trade: the **known** risk of the vulnerability in the version the operator is currently sitting on. The right amount of aging is not a constant — it is a balance:

- **Wait longer** → more confidence the new version is clean, but more continued exposure to the known vuln.
- **Adopt sooner** → clears the known vuln faster, but the operator adopts an un-vetted release.

The fulcrum is the **profile of the vulnerability the operator is currently exposed to** — severity × exploitability × actual reachability. A known _Critical, remotely-exploitable, runtime_ vuln in the current version justifies adopting even a freshly-published fix: the known present risk dominates the unknown fresh-release risk. A _Moderate, dev-only, no-attacker-input_ vuln does not: the unknown fresh-release risk dominates, and the operator should wait the full soak.

`dry-aged-deps` cannot express this today. Its `min-age` gate is unconditional, and `--severity` / `prodMinSeverity` / `devMinSeverity` only govern how clean the candidate must be — not whether the project is currently exposed and how badly. The operator's escape hatch is a global `--min-age=0`, which is a blunt instrument: it relaxes aging for every package, not only the one with the live exposure.

This story adds a new `exposure-aware-soak` modifier that consumes the already-fetched `npm audit --json` payload from `src/check-vulnerabilities.js`, maps each installed package's current-exposure max-severity onto a locked policy table, and produces a per-package effective soak. The modifier is then consumed by the existing age-gate filter in `src/filter-by-age.js` / `src/apply-filters.js`. The behaviour is opt-in via a new `--exposure-aware-soak` CLI flag (and `"exposure-aware-soak": true` config key). When opt-in, the safe-updates output surface marks which entries (if any) were age-permitted only because of the exposure-aware modifier, so operators see exactly which decisions diverged from the unconditional soak.

The locked policy table is the substance of RFC-002 §Summary, ratified at capture (user direction, 2026-05-30 Step 2.5 surfacing — commit `baee7a1` on P014). The default-OFF stance is the realization of JTBD-006 (Trust default policy): non-opt-in users see no change to the soak's behaviour until a v2 ADR ratifies the default flip.

## User Story

**Format**: So that I can pull the fix for a Critical vulnerability I am currently exposed to without waiting out the full soak _and_ without relaxing the soak globally for every other package, as a project maintainer, I want `dry-aged-deps --exposure-aware-soak` to shorten the per-package soak in proportion to my current-exposure severity per a locked policy table — Critical drops to a 0-day floor; High halves the soak; Moderate / Low / None hold the default.

**Load-bearing JTBD anchors**:

- **JTBD-001 (See safe updates)** — extends the "see only updates aged past a recent-release window and free of known vulnerabilities" surface to the inverse case: a fix is _too fresh_ to clear the unconditional soak, but the current exposure justifies surfacing it as actionable anyway. Without this story, JTBD-001's output is a strict subset of "fixes aged past `min-age`" — silent on the cell where the operator is exposed and the fix is fresh. After: the output includes those fixes when (a) the operator has opted in and (b) the exposure severity warrants. The job description's "I want to see only updates that have aged past a recent-release window" reads now as "aged past the **effective** recent-release window, where Critical exposure tightens that window to 0 days."

- **JTBD-006 (Trust default policy)** — preserved by the default-OFF v1 stance. JTBD-006's first desired outcome ("out-of-the-box behaviour rejects very recent releases") is unchanged for non-opt-in users: the default `min-age` still applies unconditionally. JTBD-006's third desired outcome ("defaults change only across major versions, with the change called out in the changelog") gates the v2 default flip: graduating `--exposure-aware-soak` from opt-in to opt-out is a v2 ADR + changelog entry per RFC-002 §Reassessment, not a quiet flip.

**INVEST Criteria Compliance**:

- **Independent**: Builds on the existing `npm audit --json` payload already consumed by `src/check-vulnerabilities.js` and the existing age-gate machinery in `src/filter-by-age.js`. No new external tooling, no new vendor relationship.
- **Negotiable**: Default-OFF for v1; the opt-in flag and config key give operators control. The locked policy table is the substance of RFC-002 §Summary; per-org overrides (e.g. "in my org, High → 0.25 × default") are explicitly v2 territory per RFC-002 §Out-of-Scope.
- **Valuable**: Closes a documented self-inflicted-risk gap (P014) — the global `--min-age=0` workaround currently trades away the soak for every package, not just the one with the live exposure. The modifier surgically scales only the affected package(s).
- **Estimable**: Reuses existing audit-cross-reference and version-time machinery. New code is the pure policy function, the per-package modifier-application, one CLI flag, one config key, and per-format formatter extensions for the "via exposure modifier" tag.
- **Small**: One new module (`src/exposure-soak-modifier.js`) or folded into `src/apply-filters.js`; one filter-pipeline wire point; three formatter extensions; one CLI flag; one config key.
- **Testable**: Detection rule is mechanical (max-severity lookup → policy table → effective soak ms). The locked policy table gives concrete test inputs (Critical → 0; High → ½ × default; Moderate/Low/None → 1.0 × default). Live-case regression test pins a Critical-exposure scenario with a freshly-published fix.

## Acceptance Criteria

- [ ] **REQ-EXPOSURE-AUDIT-INGEST**: For each installed package version, read its current advisory exposure from the already-fetched `npm audit --json` payload (the same payload `src/check-vulnerabilities.js` already consumes). Does NOT re-invoke `npm audit`. The exposure read is per-installed-version, not per-candidate-version — the policy applies to the version the operator is currently sitting on, not to the fix.
- [ ] **REQ-EXPOSURE-SEVERITY-EXTRACT**: For each installed package with one or more current advisories, compute its max-severity exposure band — the highest single severity present (Critical > High > Moderate > Low). Single-severity input only in v1; dual-severity composition (sum / max-with-cap / dependency-weighted) is explicitly v2 territory per RFC-002 §Out-of-Scope.
- [ ] **REQ-EXPOSURE-POLICY-TABLE**: Apply the locked severity → soak-modifier policy table verbatim. The table MUST be the literal RFC-002 §Summary lock — drift requires an explicit policy-change ADR per RFC-002 §Summary.

  | Current exposure severity | Soak modifier                        |
  | ------------------------- | ------------------------------------ |
  | Critical                  | **0.0 × default soak** (0-day floor) |
  | High                      | **0.5 × default soak**               |
  | Moderate                  | 1.0 × default soak (unchanged)       |
  | Low                       | 1.0 × default soak (unchanged)       |
  | None                      | 1.0 × default soak (unchanged)       |

- [ ] **REQ-EXPOSURE-EFFECTIVE-SOAK**: Effective soak for a given installed package = `min-age × modifier`. Composes with the operator's existing `--min-age=<N>` flag — the modifier scales whatever soak the operator has set, NOT a hard-coded 7 days. A `--min-age=14` operator under Critical exposure sees the same 0-day floor; a High-exposure operator sees `14 × 0.5 = 7` days; an unconditional `--min-age=0` operator sees no change (the floor and the operator's input agree).
- [ ] **REQ-EXPOSURE-PER-PACKAGE-APPLY**: The modifier applies per-installed-package, not globally. A package with Critical exposure shortens to 0-day floor; a sibling package with Moderate exposure (or no current exposure at all) preserves the full operator-set `min-age`. This is the surgical alternative to `--min-age=0` — it avoids the blunt global relaxation that currently forces operators to trade away the soak for every other package.
- [ ] **REQ-EXPOSURE-CLI-FLAG**: Add `--exposure-aware-soak` CLI flag in `src/cli-options.js`. **Default-OFF for v1**. Parsed boolean. When opt-in, the orchestrator invokes the modifier between vulnerability-fetch and age-gate; when opt-out (the default), the existing unconditional soak path runs unchanged — zero behaviour change for non-opt-in users. The opt-in default preserves JTBD-006 trust-default-policy contract on the v1 minor.
- [ ] **REQ-EXPOSURE-CONFIG-KEY**: Accept `"exposure-aware-soak"` (boolean) key in `.dry-aged-deps.json` via `src/config-loader.js`. Default-OFF if absent. CLI flag overrides config per the existing precedence rules.
- [ ] **REQ-EXPOSURE-REPORT-MODIFIED**: When opt-in, the safe-updates output surface marks which entries (if any) were age-permitted only because of the exposure-aware modifier — i.e. entries that WOULD have failed the unconditional age gate but DID pass the effective (modified) age gate. Mark format mirrors the ADR-0018 / RFC-001 short-vocabulary precedent: a `via-exposure-modifier` tag in JSON / XML; a star + footnote (e.g. `* via exposure modifier: Critical exposure → 0-day floor applied`) in the table render. Transparency surface: the operator sees exactly which decisions diverged from the unconditional soak and why.
- [ ] **REQ-EXPOSURE-TABLE**: Table output marks exposure-modified rows with a trailing star + footnote. Existing columns and rows are unchanged.
- [ ] **REQ-EXPOSURE-JSON**: JSON output gains a per-row `viaExposureModifier` field of shape `{ severity: "critical" | "high", baseSoakDays: <N>, effectiveSoakDays: <N>, advisories: [<id>] }` for rows that were age-permitted only by the modifier. Field is absent on rows that would have passed the unconditional soak anyway. Existing `rows` / `packages` / `unfixable` / `overridesHygiene` fields and their shapes are unchanged.
- [ ] **REQ-EXPOSURE-XML**: XML output gains a per-row `<viaExposureModifier>` element with the same shape as the JSON field. XML schema remains backward-compatible — the element is additive and consumers that ignore unknown elements continue to operate unchanged.
- [ ] **REQ-EXPOSURE-EXIT-CODE-UNCHANGED**: Exit-code semantics from ADR-0003 / ADR-0004 are unchanged. The modifier affects WHICH safe updates surface in the safe-update set, not the `--check` exit-1 trigger condition. `--check` exits 1 when `summary.safeUpdates > 0` OR `summary.overridesWithSafeUpgrade > 0` (per RFC-001's narrow extension); the modifier shifts membership of `summary.safeUpdates` but does NOT add a new exit-1 trigger. An opt-in user sitting on a Critical exposure sees the fix join `safeUpdates` (because the 0-day floor applies); `--check` then exits 1 via the existing trigger.
- [ ] **REQ-EXPOSURE-SEVERITY-FILTER-COMPOSITION**: Composes orthogonally with existing `--severity` / `prodMinSeverity` / `devMinSeverity` filters. Those flags gate the **cleanness of the candidate version** (a candidate with a Moderate-and-above advisory is excluded from `safeUpdates` regardless of how old it is). This modifier scales the **soak for the installed version's exposure** (the version the operator currently has, not the candidate). The two axes are independent and apply in sequence: (1) the candidate must pass cleanness, (2) the candidate must pass effective age. A candidate that is itself vulnerable still fails (1) even if Critical current exposure would otherwise relax the soak — the modifier does not waive the cleanness gate.
- [ ] **REQ-EXPOSURE-AUDIT-RESOLVE-RESPECT**: Advisories excluded in `audit-resolve.json` are filtered out of the severity-extract input. An excepted advisory does NOT contribute to the installed-package's max-severity exposure band — the operator has explicitly marked it "known and accepted." Mirrors the ADR-0008 / ADR-0018 / RFC-001 exception-respect symmetry.
- [ ] **REQ-EXPOSURE-PURE-FUNCTION**: The severity → modifier mapping function is a pure function (no I/O, no audit-payload coupling): `(severity, baseSoakMs) → effectiveSoakMs`. The orchestrator handles audit-payload parsing, max-severity computation, and per-package application separately. Unit tests target the pure function with the policy table inputs; integration tests exercise the orchestrator with realistic audit-payload fixtures.
- [ ] **REQ-EXPOSURE-REASON-VOCABULARY**: When a row is marked exposure-modified, the reason field uses a fixed short vocabulary so consumers can pattern-match: `via-exposure-modifier: critical → 0-day floor`, `via-exposure-modifier: high → halved soak (<N>d)`. Mirrors the ADR-0018 / RFC-001 short-reason vocabulary pattern.
- [ ] **REQ-EXPOSURE-OFF-BY-DEFAULT-PRESERVED**: With `--exposure-aware-soak` absent and `"exposure-aware-soak"` absent from config, the modifier is bypassed entirely — the orchestrator does NOT compute max-severity, does NOT consult the policy table, and the formatter does NOT emit the `viaExposureModifier` field/element. Existing non-opt-in users see byte-identical output to the pre-RFC-002 surface (modulo the unrelated RFC-001 / ADR-0018 surfaces already shipped).

- **REQ-EXPOSURE-AUDIT-INGEST**: New module `src/exposure-soak-modifier.js` accepts the already-loaded `npm audit --json` payload (passed in by `src/print-outdated.js` — does not re-invoke `npm audit`).

## Story Notes

**Gall's Law Compliance**: Adds one new module + a per-row annotation in three output paths + one CLI flag + one config key. Reuses existing `npm audit --json` ingestion, age-gate, audit-resolve exception-lookup, and output-formatter machinery. No new external dependency, no new vendor relationship, no new authentication surface.

**Non-breaking nature**: The modifier is **default-OFF for v1**. Non-opt-in users see exactly the pre-RFC-002 behaviour — the orchestrator skips the modifier invocation, the formatters omit the annotation, and snapshot tests pin byte-identical output for the default-OFF case (REQ-EXPOSURE-OFF-BY-DEFAULT-PRESERVED). Opt-in users see a new per-row annotation only on rows where the modifier diverged from the unconditional soak. JSON and XML schemas gain a new per-row optional field that consumers ignoring unknown fields continue to operate unchanged on. **This is not a breaking change**; no major-version bump is required. The changelog entry SHOULD call out the new opt-in surface so operators know the flag exists.

**Default-OFF rationale (JTBD-006 load-bearing)**: JTBD-006's "defaults change only across major versions, with the change called out in the changelog" gates the v2 default-flip. Shipping `--exposure-aware-soak` opt-out in v1 would silently change the soak behaviour for every non-opt-in user — a JTBD-006 violation on the minor. The opt-in path lets operators dogfood the policy with explicit consent; the v2 default-flip carries its own ADR + changelog entry per RFC-002 §Reassessment graduation triggers. JTBD-006's first desired outcome ("out-of-the-box behaviour rejects very recent releases") is preserved unchanged in v1.

**JTBD-001 extension rationale (load-bearing)**: JTBD-001's current realization shows "updates that have aged past a recent-release window and are free of known vulnerabilities." Without this story, JTBD-001 is silent on the cell where the operator is currently exposed and the fix is fresh — exactly the highest-stakes case the tool could help with. After this story (opt-in), JTBD-001's output extends to include those fixes when the exposure severity warrants. The "recent-release window" in the job description reads as the **effective** window after the modifier — Critical exposure tightens it to 0 days; High exposure halves it.

**Locked policy provenance**: The Critical → 0 / High → 0.5 / Moderate–None → 1.0 table is the substance of RFC-002 §Summary, locked at capture via user direction (P014 Fix Strategy section, 2026-05-30 Step 2.5 surfacing — commit `baee7a1`). Per RFC-002 §Summary: "The RFC may extend this shape with additional bands but MUST NOT regress the two locked points (critical → 0, high → 0.5 × default) without an explicit policy-change ADR." This spec inherits that lock — drift on the two locked points requires an ADR per the RFC's discipline.

**Boundary with `--severity` filter**: The existing `--severity` / `prodMinSeverity` / `devMinSeverity` flags gate the cleanness of the candidate version (a candidate carrying a Moderate-or-above advisory is excluded from `safeUpdates`). This modifier scales the soak for the installed version's current exposure. The two axes are orthogonal and apply in sequence — the cleanness gate runs before the age gate. A candidate that is itself vulnerable still fails the cleanness gate even if Critical current exposure would otherwise relax the soak. The modifier does NOT waive the cleanness gate; it only relaxes the time-on-registry gate.

**Boundary with `--min-age` flag**: The modifier scales whatever `min-age` the operator has set — it does NOT hard-code a 7-day base. A `--min-age=14` operator under Critical exposure sees the same 0-day floor (0.0 × 14 = 0); a High-exposure operator sees `14 × 0.5 = 7` days. An operator who has already chosen `--min-age=0` sees no change — the modifier and the operator's input agree at the floor.

**Boundary with overrides hygiene (RFC-001)**: The exposure-aware modifier reads `npm audit --json` for installed-package current exposure. The overrides-hygiene surface (RFC-001) reads the same audit payload for override-pin cross-reference. The two surfaces consume the same data but produce independent annotations — an installed-package vulnerable transitive can also have an override pin, and both surfaces will surface their respective findings independently. No interaction beyond the shared audit-payload ingest.

**Boundary with unfixable surface (ADR-0018)**: ADR-0018 surfaces known-vulnerable-but-unfixable packages — i.e. packages where no fix exists or `fixAvailable: false`. This modifier applies to packages where a fix DOES exist but is fresher-than-soak. The two surfaces are mutually exclusive on the `fixAvailable` axis — no row appears in both. The modifier does NOT touch unfixable-surface logic.

**Exit-code rationale (REQ-EXPOSURE-EXIT-CODE-UNCHANGED)**: Per RFC-002 §In-Scope item 6 — the modifier shifts membership of `safeUpdates` but does NOT add a new exit-1 trigger. ADR-0003 + ADR-0004 contracts are preserved. The existing trigger fires when an exposure-modified row joins `safeUpdates` — an opt-in user under Critical exposure sees the fix surface as `safeUpdate` and `--check` exits 1 via the existing path. No ADR amendment required.

**Audit-resolve respect rationale (REQ-EXPOSURE-AUDIT-RESOLVE-RESPECT)**: An advisory excepted in `audit-resolve.json` is the maintainer's explicit "known and accepted" mark. Allowing it to drive a soak-shortening modifier would override the operator's explicit acceptance with the modifier's "this exposure warrants haste" judgment. The operator's acceptance wins — the excepted advisory is filtered out of the severity-extract input and the modifier sees the installed package as if the exception had no exposure on that axis. ## v2 Graduation Triggers

Mirrors RFC-002 §Reassessment verbatim so the triggers are discoverable from the spec itself (not just from the RFC). Drift between this section and RFC-002 §Reassessment is a policy violation — the RFC is the source of record; this section is the spec-local restatement.

v1 ships with default-OFF. v2 default-flip (opt-in → opt-out) requires its own ADR. Default-flip graduation triggers:

1. `--exposure-aware-soak` enabled in CI for ≥ 1 calendar month with no operator-reported regression of trust/safety expectations.
2. No locked-policy violations observed (Critical → 0 / High → 0.5 hold across opt-in usage).
3. Evidence (in retro logs or operator feedback) that the policy materially reduced operator override frequency (`--min-age=0`-by-hand cases) without surfacing previously-hidden risk.

If any trigger fails or evidence accumulates that the locked policy needs amendment (e.g. operators consistently want a different Critical floor), the v2 ADR records the policy change before the default flips. The v2 ADR + changelog-entry pairing realises JTBD-006's "defaults change only across major versions, with the change called out in the changelog" outcome.
